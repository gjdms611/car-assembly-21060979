## 배경 및 목적

기존 자동차 조립 시뮬레이터(`assembly.py`)는 절차지향 방식으로 작성되어 있고, 아래 문제로 유지보수와 확장이 어려움. 본 리팩토링의 목적은 **기능/동작(콘솔 UI, 비즈니스 규칙)은 그대로 유지**하면서 내부 구조만 개선하는 것.

## 차량 제조 순서

1. 자동차 타입을 선택한다. 세단, SUV(에스-유-브이), 트럭 (향후 타입 추가 가능성 있음)
2. 자동차에 들어갈 부품을 선택한다. 엔진, 제동장치, 조향장치 (세가지 부품을 각각 선택해야 함)
3. 완성된 차량을 테스트한다. 선택한 부품이 자동차 타입에 사용 가능한지 검사

### 제한조건 1
- 제동장치에 Bosch 제품을 사용했다면, 조향장치도 Bosch 제품을 사용해야한다. (타사 제품과 호환되지 않는다.)

### 제한조건 2
- Continental은 Sedan용 제동장치를 만들지 않는다. (-> 세단에 Continental 제품 사용 불가)
- 도요타는 SUV용 엔진을 만들지 않는다.
- WIA는 Truck용 엔진을 만들지 않는다.
- Mando는 Truck용 제동장치(brake System)을 만들지 않는다.

## 기존에 사용중인 시스템의 아쉬운 점

- 절차지향식 코드로, 유지보수가 어려운 구조
- 안전하지 않은 문법들이 사용
- 확장성이 고려되지 않음
- 유닛테스트가 없음

## 리팩토링 요구사항

### 동작 유지 (변경 금지)
- 콘솔 메뉴 텍스트, 한글 안내 메시지, 입력 처리 방식(숫자 입력, `0`으로 뒤로가기, `exit` 종료)은 리팩토링 전후 완전히 동일해야 함
- 기존 호환성 규칙(제한조건 1, 2)의 판정 결과는 동일해야 함

### 테스트
- pytest 기반 유닛테스트 도입
- 호환성 규칙(제한조건 1, 2) 및 상태 전이(단계 이동, 뒤로가기)에 대한 테스트 커버리지 확보

### 확장성
- 새 차량 타입/부품/호환성 규칙 추가 시 "코드 수정"은 허용됨 (설정 파일만으로 확장할 필요는 없음)
- 다만 규칙이 한 곳에서만 관리되어, 규칙 추가/변경 시 여러 파일(또는 여러 함수)을 중복 수정할 필요는 없어야 함

### Out of scope
- 새로운 차량 타입/부품 실제 추가 (구조 개선만 대상, 신규 기능 추가 아님)
- UI/UX 변경, 텍스트 다국어화
